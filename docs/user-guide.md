# Financial Journey: руководство пользователя

Документ описывает текущую MVP-версию приложения Financial Journey: как войти, какие есть типы аккаунтов, что может каждая роль и по каким правилам проходит партия.

Приложение открывается в браузере:

- Web: `http://localhost:3000`
- API health check: `http://localhost:4000`

## 1. Основные понятия

В системе есть два разных уровня ролей:

- Тип аккаунта в системе: `USER`, `HOST`, `ADMIN`.
- Роль участника внутри конкретной игровой комнаты: `HOST`, `PLAYER`, `BANKER`, `OBSERVER`.

Это не одно и то же. Например, аккаунт `HOST` может создавать комнаты, а роль `HOST` внутри комнаты означает ведущего этой конкретной партии. Обычный пользователь может войти в комнату как игрок, банкир или наблюдатель, если комната еще не запущена.

## 2. Вход и регистрация

1. Откройте `http://localhost:3000`.
2. Если аккаунта нет, нажмите `Зарегистрироваться`.
3. Укажите имя игрока, email и пароль от 8 символов.
4. После регистрации система автоматически войдет в личный кабинет.
5. Для повторного входа используйте email и пароль на странице `Вход`.
6. Для выхода нажмите `Выйти` в правом верхнем углу.

Если аккаунт заблокирован или удален администратором, войти под ним нельзя.

## 3. Типы аккаунтов

### Пользователь: `USER`

Обычный игрокский аккаунт.

Может:

- регистрироваться и входить в личный кабинет;
- видеть свой профиль, статистику и историю партий;
- входить в открытые комнаты по коду;
- выбрать роль в комнате: `Игрок`, `Банкир` или `Наблюдатель`;
- участвовать в партии, если вошел как `Игрок`;
- пользоваться чатом и смотреть журнал действий в доступной комнате.

Не может:

- создавать новые игры;
- запускать или удалять игры;
- добавлять других пользователей в игру;
- управлять аккаунтами других пользователей.

### Ведущий: `HOST`

Аккаунт для человека, который организует партии.

Может все, что может `USER`, а также:

- создавать игровые комнаты;
- задавать название партии и лимит игроков от 2 до 6;
- получать код комнаты для приглашения участников;
- добавлять пользователей в комнату до старта игры;
- назначать добавленным пользователям роль `Игрок`, `Банкир` или `Наблюдатель`;
- запускать созданную им игру;
- удалять созданную им игру.

Важно: создатель комнаты становится ведущим этой комнаты и не занимает место игрока. Чтобы начать партию, в комнате должно быть минимум два участника с ролью `Игрок`.

### Администратор: `ADMIN`

Аккаунт для управления системой.

Может все, что может `HOST`, а также:

- видеть административный блок `Администрирование аккаунтов`;
- создавать аккаунты пользователей, ведущих и администраторов;
- менять системную роль аккаунта: `USER`, `HOST`, `ADMIN`;
- блокировать и разблокировать аккаунты;
- удалять аккаунты из системы;
- видеть и администрировать игры в системе.

Защита от случайной потери доступа:

- администратор не может заблокировать сам себя;
- администратор не может удалить сам себя;
- администратор не может понизить собственную роль ниже `ADMIN`.

Удаление аккаунта в MVP работает как мягкое удаление: аккаунт получает статус `DELETED`, скрывается из активного списка и больше не может войти.

## 4. Статусы аккаунтов

- `ACTIVE`: аккаунт активен, пользователь может входить и играть.
- `BLOCKED`: аккаунт заблокирован, вход запрещен до разблокировки.
- `DELETED`: аккаунт удален, вход запрещен.

## 5. Роли внутри игровой комнаты

### Ведущий комнаты: `HOST`

Назначается автоматически создателю комнаты.

Может:

- видеть комнату;
- добавлять участников до старта;
- запускать партию;
- удалить партию.

Ведущий комнаты не ходит по полю, не получает профессию и не имеет финансового отчета игрока.

### Игрок: `PLAYER`

Основной участник партии.

Может:

- получить профессию при старте партии;
- ходить по полю в свой ход;
- бросать кубик;
- получать зарплату и cashflow;
- тянуть карточки;
- покупать сделки;
- брать кредит;
- видеть свой финансовый отчет;
- выйти из крысиных бегов при выполнении финансового условия.

Только роль `PLAYER` может выполнять игровые финансовые действия.

### Банкир: `BANKER`

В текущем MVP это наблюдательная роль для помощи ведущему и игрокам.

Может:

- войти в комнату;
- смотреть состояние партии;
- читать журнал действий;
- пользоваться чатом.

Автоматизированных специальных банковских прав в интерфейсе пока нет: кредиты и пересчет финансов выполняет сервер.

### Наблюдатель: `OBSERVER`

Роль для зрителей или учеников.

Может:

- войти в комнату;
- смотреть поле, игроков, журнал и чат;
- не участвует в ходах и финансовых действиях.

## 6. Личный кабинет

После входа пользователь попадает в личный кабинет.

В кабинете отображаются:

- системная роль и статус аккаунта;
- количество партий;
- количество побед;
- количество выходов из крысиных бегов;
- средний cashflow;
- список активных партий пользователя;
- список открытых комнат;
- история партий.

Для администратора дополнительно отображается блок управления аккаунтами.

## 7. Создание и вход в комнату

### Создание комнаты

Доступно только `HOST` и `ADMIN`.

1. В личном кабинете найдите блок `Новая комната`.
2. Укажите название партии.
3. Укажите максимум игроков: от 2 до 6.
4. Нажмите `Создать`.
5. После создания откроется игровая комната.
6. Скопируйте код комнаты и передайте участникам.

### Добавление пользователей ведущим

Доступно ведущему комнаты и администратору, пока партия находится в статусе `WAITING`.

1. Откройте комнату.
2. В блоке `Ведущий` укажите email пользователя.
3. Выберите роль: `Игрок`, `Банкир` или `Наблюдатель`.
4. Нажмите `Добавить в игру`.

Добавить можно только активный аккаунт. Если пользователь заблокирован или удален, система не добавит его в игру.

### Самостоятельный вход по коду

1. В личном кабинете введите код комнаты.
2. Выберите роль: `Игрок`, `Банкир` или `Наблюдатель`.
3. Нажмите `Войти`.

Войти можно только в комнату со статусом `WAITING`. После старта новые участники через форму входа не добавляются.

## 8. Статусы игры

- `WAITING`: комната создана, идет набор участников.
- `IN_PROGRESS`: партия запущена, идут ходы игроков.
- `ENDED`: партия завершена.
- `CANCELLED`: партия удалена или отменена.

Для старта нужно минимум два участника с ролью `PLAYER`.

## 9. Как проходит партия

### Подготовка

При запуске партии сервер:

1. Проверяет, что в комнате есть минимум два игрока.
2. Случайно назначает каждому игроку профессию из справочника.
3. Создает финансовый отчет каждого игрока.
4. Переносит игроков на малый круг.
5. Устанавливает первый ход и первый раунд.

Профессия задает стартовые значения:

- зарплата;
- пассивный доход;
- обязательные расходы;
- начальные сбережения;
- долги и ежемесячные платежи;
- стоимость содержания ребенка.

### Ход игрока

1. Активный игрок нажимает `Бросить кубик`.
2. Сервер бросает кубик от 1 до 6.
3. Сервер передвигает игрока по малому кругу.
4. Если игрок прошел старт или попал на `Paycheck`, он получает месячный cashflow.
5. Сервер обрабатывает клетку, на которую попал игрок.
6. Финансовый отчет пересчитывается автоматически.
7. Если пассивный доход стал больше расходов, игрок побеждает и партия завершается.
8. Ход переходит к следующему игроку.

Клиент только отправляет намерение игрока. Все расчеты делает backend, поэтому состояние партии считается сервером авторитетным.

## 10. Малый круг

В MVP малый круг состоит из 24 клеток:

| Клетка | Что происходит |
| --- | --- |
| `Paycheck` | Игрок получает месячный cashflow. |
| `Small Deal` | Игрок тянет карточку малой сделки. |
| `Big Deal` | Игрок тянет карточку большой сделки. |
| `Market` | Игрок тянет рыночную карточку. |
| `Doodad` | Игрок тянет карточку расходов, эффект применяется автоматически. |
| `Charity` | Если хватает денег, игрок жертвует 10% общего дохода. |
| `Baby` | У игрока появляется ребенок, максимум 3. Расходы увеличиваются. |
| `Downsized` | Игрок платит сумму месячных расходов и пропускает 2 хода. |

## 11. Карточки

В игре используются карточки из seed-данных:

- `small_deal`: малые сделки;
- `big_deal`: большие сделки;
- `market`: рынок;
- `doodad`: непредвиденные расходы;
- `fast_track`: сделки быстрого круга;
- `dream`: мечты.

Когда игрок попадает на клетку сделки или нажимает кнопку карточки, сервер случайно выбирает активную карточку нужного типа и записывает событие в журнал.

### Small Deal и Big Deal

Сделки можно покупать, если у игрока достаточно наличных для первоначального взноса.

При покупке:

1. Из наличных списывается первоначальный взнос.
2. В активы добавляется купленный объект.
3. Cashflow от актива прибавляется к пассивному доходу.
4. Финансовый отчет пересчитывается.
5. Событие покупки записывается в журнал.

Если денег на первоначальный взнос не хватает, покупка не выполняется.

### Market

Рыночные карточки отображаются и записываются в журнал. Полная автоматизация продажи активов по рынку в текущем MVP еще не вынесена в интерфейс.

### Doodad

Doodad-карточки применяются автоматически:

- если карточка требует наличие детей, а детей нет, эффект не применяется;
- если есть денежный эффект, он изменяет наличные игрока;
- событие записывается в журнал.

### Fast Track и Dream

Справочники карточек быстрого круга и мечт уже загружены, но в текущем
режиме партия завершается сразу после достижения финансовой свободы.

## 12. Финансовый отчет

Финансовый отчет игрока пересчитывается сервером после действий, которые меняют деньги, активы, долги или расходы.

Основные поля:

- `Cash`: наличные деньги.
- `Cashflow`: месячный денежный поток.
- `Passive`: пассивный доход.
- `Expenses`: общие расходы.
- `Активы`: купленные сделки, приносящие или не приносящие cashflow.
- `Долги`: обязательства и ежемесячные платежи.

Формула MVP:

```text
Пассивный доход = базовый пассивный доход профессии + cashflow активов
Общие расходы = базовые расходы + платежи по долгам + расходы на детей
Общий доход = зарплата + пассивный доход
Месячный cashflow = общий доход - общие расходы
```

## 13. Кредиты и долги

Игрок может взять банковский кредит в блоке `Действия`.

При взятии кредита:

- наличные увеличиваются на сумму кредита;
- создается долг `Bank loan`;
- ежемесячный платеж по умолчанию равен 10% суммы кредита;
- расходы и cashflow пересчитываются.

Погашение кредита поддержано backend-логикой, но отдельная кнопка погашения в текущем MVP еще не вынесена в интерфейс.

## 14. Победа и ограничение времени

Игрок побеждает, когда:

```text
пассивный доход > общие расходы
```

После выполнения условия сервер:

- фиксирует дату и время победы;
- завершает партию;
- записывает событие `game:ended`;
- обновляет статистику в личном кабинете.

При создании комнаты ведущий задаёт лимит от 15 до 240 минут. По умолчанию
партия длится 90 минут. Если за это время никто не выполнил условие победы,
сервер завершает партию без победителя.

## 15. Журнал, чат и история

Каждое важное действие сохраняется как событие:

- создание игры;
- добавление игрока;
- старт игры;
- бросок кубика;
- движение по полю;
- получение cashflow;
- вытягивание карточки;
- покупка сделки;
- взятие кредита;
- погашение кредита;
- выход из крысиных бегов;
- удаление игры.

Журнал действий виден в игровой комнате. Чат работает в реальном времени для участников комнаты.

История партий отображается в личном кабинете. Данные replay сохраняются через события `game_events`; отдельный пользовательский экран replay в текущем MVP еще не завершен.

## 16. Рекомендованный порядок проведения партии

1. Администратор создает аккаунты или участники регистрируются сами.
2. Администратор назначает одному пользователю роль `HOST`, если нужен ведущий.
3. Ведущий создает комнату.
4. Ведущий добавляет игроков или отправляет им код комнаты.
5. Игроки заходят в комнату до старта.
6. Ведущий проверяет, что есть минимум два игрока.
7. Ведущий нажимает `Начать партию`.
8. Игроки ходят по очереди.
9. Игроки покупают сделки, берут кредиты и следят за отчетом.
10. Цель партии: до истечения времени добиться условия `пассивный доход > общие расходы`.

## 17. Что делать при проблемах

### Не получается войти

Проверьте:

- правильность email и пароля;
- не заблокирован ли аккаунт;
- не удален ли аккаунт.

### Нет кнопки создания игры

Создавать игры могут только `HOST` и `ADMIN`. Попросите администратора изменить роль аккаунта.

### Не получается стартовать игру

Проверьте:

- вы ведущий этой комнаты или администратор;
- игра находится в статусе `WAITING`;
- в комнате есть минимум два `PLAYER`.

### Игрок не может бросить кубик

Проверьте:

- игра уже запущена;
- сейчас ход именно этого игрока;
- пользователь вошел как `PLAYER`, а не как `BANKER` или `OBSERVER`;
- игрок не находится в состоянии `Downsized` с пропуском хода.

### Сделка не покупается

Чаще всего причина в том, что не хватает наличных для первоначального взноса.

## 18. Ограничения текущего MVP

Текущая версия уже позволяет проводить базовую партию на малом круге, но часть полного настольного процесса еще не автоматизирована:

- быстрый круг и победа через мечту в текущем режиме не используются;
- мечты и автоматическая победа пока не завершены в интерфейсе;
- продажа активов по рыночным карточкам пока не вынесена в интерфейс;
- погашение кредита поддержано backend-логикой, но не вынесено отдельной кнопкой в игровой комнате;
- роль `BANKER` пока не имеет специальных прав сверх наблюдения и чата.

Эти ограничения важны для проведения партий: если нужно строго следовать полной настольной версии, ведущий должен учитывать недостающие действия вручную до завершения следующих этапов разработки.

## 19. Архитектура проекта

Financial Journey сделан как монорепозиторий: frontend, backend, база данных и общая игровая логика лежат в одном проекте, но разделены по зонам ответственности.

### Общая структура

Основные части проекта:

- `apps/web`: браузерное приложение на Next.js и TypeScript;
- `apps/api`: backend API на NestJS;
- `packages/database`: Prisma-схема, миграции и seed-скрипты;
- `packages/shared`: общие типы, игровое поле и базовые функции движка;
- `docs`: пользовательская документация, включая это руководство.

Такое разделение позволяет держать UI, серверную бизнес-логику, модель базы и общие игровые правила отдельно, но использовать единые типы и справочники.

### Frontend: `apps/web`

Frontend работает в браузере и отвечает за интерфейс:

- страницы входа, регистрации, личного кабинета, админ-панели и руководства;
- игровую комнату;
- финансовый отчет игрока;
- поле малого круга;
- чат и журнал действий;
- формы создания игры, добавления игроков, покупки сделок и взятия кредита.

Важный принцип: frontend не рассчитывает итоговое состояние игры самостоятельно. Он показывает текущее состояние, отправляет намерение пользователя на backend и получает обновленный snapshot игры.

Примеры намерений:

- игрок нажал `Бросить кубик`;
- игрок выбрал мелкую или крупную сделку;
- игрок купил карточку;
- игрок взял кредит;
- ведущий добавил пользователя в игру;
- администратор изменил карточку или аккаунт.

### Backend: `apps/api`

Backend на NestJS является авторитетным источником правил и расчетов.

Он отвечает за:

- регистрацию и вход;
- проверку JWT;
- проверку ролей `USER`, `HOST`, `ADMIN`;
- создание, запуск и удаление игр;
- добавление пользователей в комнату;
- бросок кубика;
- движение по полю;
- обработку клеток;
- вытягивание карточек;
- покупку сделок;
- кредиты;
- пересчет финансового отчета;
- запись истории событий;
- отправку realtime-обновлений через Socket.IO.

Клиент не может сам назначить себе деньги, активы, позицию или результат броска. Он только отправляет запрос. Сервер проверяет права пользователя, текущее состояние игры и правила, затем меняет данные в базе.

### Realtime-синхронизация

Игровая комната использует Socket.IO.

Основная схема такая:

1. Пользователь открывает игру в браузере.
2. Frontend подключается к Socket.IO namespace игры.
3. Пользователь выполняет действие, например `player:roll_dice`.
4. Backend проверяет действие и выполняет расчет.
5. Backend записывает события в `game_events`.
6. Backend собирает новый snapshot игры.
7. Все участники комнаты получают обновление `state:update`.

Благодаря этому игроки, ведущий и наблюдатели видят одно и то же состояние без ручного обновления страницы.

### База данных и Prisma

База данных PostgreSQL хранит и текущее состояние, и историю игры.

Ключевые таблицы:

- `users`: аккаунты пользователей;
- `games`: игровые комнаты и статус партии;
- `game_players`: участники конкретной игры и их роли в комнате;
- `professions`: профессии игроков;
- `cards`: карточки сделок, рынка, всякой всячины, быстрого круга и мечт;
- `player_financial_state`: текущий финансовый отчет игрока;
- `player_assets`: активы игрока;
- `player_liabilities`: долги игрока;
- `game_events`: журнал всех игровых действий;
- `game_chat_messages`: сообщения чата.

Prisma описывает схему базы и дает backend-у типизированный доступ к данным. Seed-скрипты загружают профессии и карточки из SQL-файлов.

### Событийная модель

Главный принцип проекта: не хранить только итоговое состояние.

При каждом важном действии создается запись в `game_events`. Например:

- игрок бросил кубик;
- игрок переместился на клетку;
- игрок получил расчетный чек;
- игрок вытянул карточку;
- игрок купил сделку;
- игрок взял кредит;
- игрок вышел из крысиных бегов.

Текущее состояние нужно для быстрой работы интерфейса, но события нужны для аудита, истории и будущего replay партии.

### Snapshot игры

Snapshot игры — это полное состояние, которое frontend получает от backend:

- данные комнаты;
- текущее положение хода;
- поле;
- список игроков;
- финансовые отчеты;
- активы и долги;
- последние события;
- сообщения чата;
- ожидаемое действие, если ход не завершен.

Frontend не восстанавливает состояние из отдельных событий прямо в браузере. Этим занимается backend. Браузер получает уже готовый snapshot и отображает его.

### Роли в архитектуре

В системе есть два уровня ролей:

- системная роль аккаунта: `USER`, `HOST`, `ADMIN`;
- роль в конкретной игре: `HOST`, `PLAYER`, `BANKER`, `OBSERVER`.

Backend проверяет оба уровня. Например, аккаунт `ADMIN` может управлять системой, но в игровой комнате он не должен отображаться как игрок, если его роль в этой комнате не `PLAYER`.

## 20. Руководство админ-панели

Админ-панель находится в личном кабинете администратора и разделена на пункты:

- `Дашборд`: текущие игры, новая комната, история игр и краткие результаты;
- `Пользователи`: управление аккаунтами;
- `Настройки > Карточки игры`: создание и редактирование карточек;
- `Настройки > Игровое поле`: просмотр структуры малого и быстрого круга;
- `Руководство`: переход к этому документу.

Админ-панель работает через защищенные API-методы. Доступ к ним есть только у аккаунтов с ролью `ADMIN`.

### Управление карточками игры

Раздел `Настройки > Карточки игры` нужен для создания и редактирования игровых карточек. Главный принцип: текст карточки объясняет правило человеку, а игровые действия должны быть описаны структурированно в полях карточки. Сервер выполняет именно структурированные действия, а не пытается каждый раз заново читать смысл из текста.

Карточка состоит из основных полей:

| Поле | Назначение |
| --- | --- |
| `cardType` | Колода карточки: малая сделка, крупная сделка, рынок, всякая всячина, быстрый круг или мечта. |
| `slug` | Уникальный технический код карточки. Нужен для seed-данных, поиска и обновлений. |
| `title` | Название карточки, которое видит игрок. |
| `bodyText` | Полный текст карточки для игрока. |
| `category` | Смысловая категория: например `stock`, `realestate`, `market`, `expense`. |
| `subcategory` | Дополнительная категория, если нужна. |
| `isActive` | Если выключено, карточка не выпадает в игре. |
| `meta` | Параметры карточки: цена, тикер акции, ипотека, первоначальный взнос и т.п. |
| `effects` | Структурированные действия, которые должен выполнить сервер. |
| `conditions` | Условия, при которых действия карточки применяются. |

### Логика работы карточек

Когда игрок вытягивает карточку, backend делает следующее:

1. Выбирает случайную активную карточку нужного типа.
2. Записывает событие `card:draw` в журнал.
3. Проверяет условия карточки из `conditions`.
4. Выполняет структурированные действия из `effects`.
5. Пересчитывает финансовый отчет игрока.
6. Записывает результат в журнал действий.

Важно: клиент только показывает карточку и отправляет намерение игрока, например купить сделку. Все расчеты выполняет backend.

### Обязательные и необязательные карточки

Есть два разных режима:

| Тип ситуации | Можно отказаться? | Как работает |
| --- | --- | --- |
| Сделка `small_deal` или `big_deal` | Да | Игрок может купить актив или отказаться от покупки. |
| `doodad` / всякая всячина | Нет | Если условие выполнено, расход или долг применяется автоматически. |
| Автоматическая рыночная/акционная карточка | Нет | Например дробление, уменьшение или обнуление акций применяется к владельцам подходящего актива. |
| Карточка с невыполненным условием | Не требуется | Если условие не выполнено, эффект не применяется и в журнал пишется причина. |

Если в тексте Doodad-карточки написано “Заплатите”, “Потратили”, “Добавьте в пассив” или “Добавьте в ежемесячные расходы”, это обязательное действие. Его нужно обязательно описать в `effects`; одного текста недостаточно.

### Список правил карточек

Поддерживаемые структурированные действия:

| `effectType` | Что делает | Основные поля |
| --- | --- | --- |
| `cash.adjust` | Меняет наличные игрока. Отрицательная сумма списывает деньги, положительная добавляет деньги. | `amountCents` |
| `cashflow.adjust` | Меняет денежный поток. Положительная сумма добавляет пассивный доход, отрицательная добавляет ежемесячный расход. | `amountCents` |
| `liability.create` | Создает долг и ежемесячный платеж. | `amountCents`, `payload.paymentCents`, `payload.type`, `payload.name` |
| `asset.quantity.multiply` | Умножает количество акций у всех владельцев указанного тикера. | `amountCents`, `meta.symbol` |
| `asset.quantity.divide` | Делит количество акций у всех владельцев указанного тикера. | `amountCents`, `meta.symbol` |
| `asset.wipeout` | Обнуляет активы с указанным тикером. | `meta.symbol` |

Старые названия эффектов тоже поддерживаются для совместимости:

| Старое название | Новое модульное действие |
| --- | --- |
| `cash_delta` | `cash.adjust` |
| `conditional_cash_delta` | `cash.adjust` с условием |
| `cashflow_delta` | `cashflow.adjust` |
| `stock_split` | `asset.quantity.multiply` |
| `stock_reverse_split` | `asset.quantity.divide` |
| `stock_wipeout` | `asset.wipeout` |

Поддерживаемые условия:

| `condType` | Что проверяет |
| --- | --- |
| `has_children` | У игрока есть хотя бы один ребенок. |
| `has_rental_realestate` | У игрока есть сдаваемая недвижимость. |
| `has_8_plex` | У игрока есть 8-квартирный дом. |

Частые `meta`-поля:

| `metaKey` | Назначение |
| --- | --- |
| `symbol` | Тикер акции или фонда, например `POP1`, `ISN`, `2BIG`. |
| `today_price` | Текущая цена акции. |
| `price` | Цена покупки или цена предложения рынка. |
| `price_min` | Нижняя граница возможной цены. |
| `price_max` | Верхняя граница возможной цены. |
| `down_payment` | Первоначальный взнос для сделки. |
| `mortgage` | Ипотека или закладная по объекту. |
| `cashflow_monthly` | Ежемесячный cashflow актива. |
| `per_child` | Если `true`, расход умножается на количество детей. |

### Как правильно создать карточку

1. Откройте `Админ-панель > Настройки > Карточки игры`.
2. Выберите фильтр нужного типа карточек.
3. Нажмите создание новой карточки.
4. Заполните `cardType`, `slug`, `title`, `bodyText`, `category`.
5. Добавьте `meta`, если карточке нужны параметры: цена, тикер, взнос, ипотека, cashflow.
6. Добавьте `effects`, которые сервер должен выполнить.
7. Добавьте `conditions`, если эффект зависит от состояния игрока.
8. Проверьте, что текст карточки и структурированные действия не противоречат друг другу.
9. Сохраните карточку.
10. Проверьте карточку в тестовой партии.

Пример обязательного расхода:

```json
{
  "effectType": "cash.adjust",
  "amountCents": -1000,
  "payload": { "mandatory": true }
}
```

Пример обязательного долга:

```json
{
  "effectType": "liability.create",
  "amountCents": 17000,
  "payload": {
    "mandatory": true,
    "type": "doodad_loan",
    "name": "Новый водный глиссер",
    "paymentCents": 340
  }
}
```

Пример расхода за каждого ребенка:

```json
{
  "meta": [{ "metaKey": "per_child", "metaValue": "true" }],
  "effects": [{ "effectType": "cash.adjust", "amountCents": -50 }],
  "conditions": [{ "condType": "has_children" }]
}
```

### Как правильно редактировать карточку

При редактировании карточки используйте такие правила:

1. Если меняете текстовое правило, сразу меняйте `effects` и `conditions`.
2. Не оставляйте обязательный расход только в тексте карточки.
3. Если карточка добавляет долг, используйте `liability.create`, а не только `cash.adjust`.
4. Если карточка меняет ежемесячные расходы или доход, используйте `cashflow.adjust`.
5. Если карточка работает с акциями, задайте `meta.symbol`.
6. Если карточка не должна выпадать, выключите `isActive`, а не удаляйте ее.
7. Не меняйте `slug` без необходимости: он используется как стабильный технический идентификатор.
8. Перед изменением карточек во время активной партии предупредите игроков: новые вытягивания будут использовать обновленную карточку, но уже записанные события истории не переписываются.

### Типовые ошибки при карточках

Чего нельзя делать:

- написать “Заплатите $500” только в `bodyText` и не добавить `cash.adjust`;
- написать “Добавьте долг $4000” и не добавить `liability.create`;
- добавить `asset.wipeout`, но забыть `meta.symbol`;
- поставить условие `has_children`, но забыть указать денежный эффект;
- сделать карточку активной, не проверив ее в тестовой игре;
- менять смысл старой карточки во время партии без предупреждения игроков.
